## 1. Editable / custom access control (`access_control`)
Make roles editable by adding per-company custom roles alongside the existing 3 global seeded roles, plus full CRUD on the role/permission matrix. DESIGN DECISION (grounded in the code): keep the existing `Role` table (a `BaseModel`, NOT `CompanyModel`) but add a NULLABLE `company_id` FK to `companies.id`. NULL = global seeded role (admin/manager/operator); non-NULL = a tenant-owned custom role. This preserves all four global `Role.code == "admin"` lookups that drive enforcement and bootstrap — dependencies.py:140 (impersonation), routers/auth.py:95 (session), services/member.py:19 (`_admin_role_id` / last-admin guard), services/admin.py:105 — because seeded rows stay company_id=NULL and remain the only rows with reserved codes. RequirePermission (dependencies.py:171) keeps working unchanged: it reads `user.role.permissions` codes regardless of who owns the role. The DB change requires dropping the GLOBAL unique index `ix_roles_code` (created in feea7ff730da_initial_schema.py:58 as unique=True) and replacing it with a composite unique on `(company_id, code)` so two companies can each have a `sales` role and seeded codes stay unique among globals. Service layer: a new `list_roles` query returns `company_id IS NULL OR company_id = <tenant>` (replacing today's unscoped global list in services/role.py); create/update/delete operate only on `company_id == tenant` rows (NEVER touch globals — guard with a 409 if a caller targets a seeded/global role), validate submitted permission codes against the seeded `permissions` table, reject reserved codes (admin/manager/operator), refuse delete when any User still references the role (FK is ondelete=RESTRICT), and write audit entries via services/_audit.write_audit with resource_type="roles". Router gains POST/PATCH/DELETE under the existing /v1/roles prefix guarded by RequirePermission("roles.write"). Frontend: convert the read-only Funções pane (settings/roles/page.tsx + PermissionMatrix.tsx) into an editable matrix — enable the existing disabled "Criar função personalizada" button (page.tsx:95-106, data-testid create-custom-role) to open a create-role sheet, and let custom-role columns toggle each domain.read/domain.write permission with optimistic save via new TanStack mutation hooks in use-roles.ts. Seeded roles render read-only (no toggles); only company-owned roles are editable/deletable.

- Permissions: roles.read (existing — reused for GET), roles.write (existing — reused to guard POST/PATCH/DELETE; no NEW permission code or seed/grant migration needed since roles.write is already seeded and granted to admin)

## 3. Demand-driven print queue + auto-netting (`print_queue`)

The legacy reference (`backend/scripts/base44/raw/AjusteEstampa.json`) confirms the key is `estampa_nome||cor` with `quantidade` (needed) and `impresso` (printed/on-hand) fields — this maps to `(print_design, product_color)` + on-hand. Reuse the existing `prints.read`/`prints.write` permission domain (already seeded in 3187f02cbc35) rather than a new domain, since print stock is conceptually part of estampas; no new permission migration is strictly required, but a new `print_stock.read`/`print_stock.write` pair is the cleaner option and is included below. Step 3: when a design's prints are dispatched (in `services/montador.send_batch_to_montador`, where `adj.prints_sent` flips to True only when all sides succeed) AND/OR when the batch transitions to PRINTED (`services/batch.transition_status`), write a `PrintStockExit` of `qty_to_print` per adjustment row, guarded by an idempotency flag so a re-send/re-transition does not double-decrement (add `stock_committed_at` to `BatchPrintAdjustment`). Step 4: a cross-batch queue view = aggregate of all `BatchPrintAdjustment` rows in batches with status OPEN/ADJUSTED where `qty_to_print > 0` and not yet sent, grouped by design×color, exposed at `GET /v1/batches/print-queue`. Step 5 frontend: enhance the Lotes detail page (`orders/batches/[id]/page.tsx`) to show needed/on-hand/to-print per design per `docs/design/pages/lotes.jsx` (the EstampaThumb + "Necessário"/"A imprimir" StatCards, "Montar"/"Montado" toggle), add a queue page/section, and a print-stock inventory screen mirroring the Stock levels UI. Upsert (delete+recreate) the row on each DONE transition; the row stays when an order is reverted to CUTTING (last computed cost kept) — re-DONE recomputes. ## 9. Turnover / "giro" report (`turnover_report`)
Add an inventory turnover / giro report as a fifth read-only aggregator, fully additive on top of the existing FEATURE-015 reports machinery (schemas/reports.py, services/reports.py, routers/reports.py, frontend reports/ components + use-reports.ts). NO new DB tables/columns/enums and NO Alembic schema change beyond a permission seed. Turnover is computed per ProductVariation (and rolled up per product spec) over the [date_from, date_to] window from the two stock-ledger tables already used by inventory_report: StockExit rows with reason=SALE give units sold in the period; StockEntry minus StockExit give on-hand. Use the same scoped()-based subquery pattern inventory_report already uses. Compute: units_sold (sum of StockExit.quantity where reason=SALE, created_at in window); on_hand_end (lifetime entries - exits up to date_to); on_hand_start (entries-exits up to date_from); average_on_hand = (on_hand_start + on_hand_end)/2 (fall back to on_hand_end when start is 0/unknown); turnover_ratio = units_sold / average_on_hand (0 when avg<=0); days_inventory_outstanding (DIO) = period_days / turnover_ratio (None/0 when turnover_ratio==0). period_days = max(1,(date_to-date_from).days). Reuse _resolve_range() for the default 30-day window. Return a list of per-variation rows (variation_id, sku, spec_code via join through ProductVariation->Product->ProductSpec, units_sold, average_on_hand, turnover_ratio, days_inventory_outstanding) sorted by turnover_ratio desc, plus aggregate totals.
